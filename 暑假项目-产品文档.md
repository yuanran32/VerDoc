# 暑假项目 · 产品文档

> 项目暂名: VerDoc / DocLens / 文档罗盘  
> 产品定位: 面向前端开发者的版本感知文档问答助手  
> 配套文档: `暑假项目-技术方案.md`  
> 最后更新: 2026-07-06

---

## 1. 产品概述

VerDoc 是一个面向前端开发者的官方文档智能问答工具。用户可以围绕 Vue 生态提出文档查询、API 用法、版本差异和迁移相关问题，系统基于官方文档检索后生成回答，并提供可点击的来源引用和明确的版本标注。

产品不追求做通用聊天机器人，而是聚焦一个高价值问题:开发者在使用前端框架时，经常被文档版本、API 变更、过期答案和 LLM 幻觉误导。VerDoc 通过版本元数据过滤、RAG 检索增强和 citation 溯源，让回答更可信、更可验证。

一句话定位:

> 一个知道文档版本、敢于拒答、并且每个结论都能追溯到官方来源的前端框架文档助手。

---

## 2. 目标用户

### 2.1 核心用户

- 使用 Vue 3、Vite、Pinia、Vue Router 的前端开发者
- 正在从 Vue 2 迁移到 Vue 3 的开发者
- 经常查官方文档、StackOverflow、博客和 AI 问答的学习者
- 准备前端面试、需要系统理解框架 API 的学生或初级开发者

### 2.2 用户特征

- 有明确技术问题，不需要闲聊
- 关注答案是否适用于当前框架版本
- 需要能够快速跳回官方文档验证结论
- 对错误答案容忍度低，尤其是 API 是否存在、版本是否支持这类问题

---

## 3. 用户痛点

| 痛点 | 典型表现 | 产品解法 |
|---|---|---|
| 文档版本混乱 | 搜到的答案可能来自 Vue 2、Vue 3.2 或 Vue 3.4，不知道是否适用 | 回答前按版本过滤文档，答案中标注适用版本 |
| LLM 容易幻觉 | 通用模型会编造不存在的 API 或给出过时写法 | 只基于检索片段回答，找不到证据就拒答 |
| 答案难验证 | AI 给出的结论没有出处，不敢直接用 | 每个关键结论带 citation，可跳转官方文档 |
| 查文档效率低 | 官方文档信息分散，需要自己翻目录和搜索 | 用自然语言提问，系统返回整理后的答案 |
| 迁移成本高 | Vue 2 到 Vue 3 的 breaking change 分散在迁移文档中 | 提供迁移建议，并引用 migration guide 来源 |

## 4. 竞品与差异化

基于 2026-07-05 的调研:

| 竞品 | 性质 | 与本产品的关系 |
|---|---|---|
| [kapa.ai](https://www.kapa.ai) | 商业产品，200+ 企业客户 | 版本感知是其招牌功能，证明该能力的商业价值；不开源 |
| [chat-langchain](https://github.com/langchain-ai/chat-langchain) | 开源，LangChain 官方 | 架构最佳参考(拒答护栏 / citation / 流式)；无版本感知 |
| [kotaemon](https://github.com/Cinnamon/kotaemon) | 开源 RAG UI | 混合检索 + rerank 的实现参考 |
| [OpenDocuments](https://github.com/joungminsung/OpenDocuments) | 开源自托管 | citation + 置信分的交互参考 |

**差异化结论:版本感知(version-aware)是「商业上已被验证、开源侧无人做成一等公民」的空白点，本产品将其作为核心卖点。** 其余能力(citation、拒答、流式)是开源社区已有最佳实践的组合，产品价值在于把它们与版本感知整合成完整体验。

## 5. 产品目标

### 5.1 短期目标

- 支持 Vue 生态官方文档问答
- 回答必须带来源引用
- 支持框架和版本选择
- 对知识库外问题明确拒答
- 提供可演示的 Web UI

### 5.2 中期目标

- 支持多版本文档检索
- 支持版本差异提示
- 支持 Vue 2 到 Vue 3 的迁移问答
- 建立评测集，量化检索命中率、答案忠实度和引用准确率

### 5.3 长期目标

- 扩展到更多前端生态，例如 React、Next.js、Nuxt
- 加入反馈闭环，收集 bad case
- 支持多轮上下文追问
- 发展为面向技术文档的通用 version-aware RAG 模板项目

## 6. 核心场景

### 场景 1: 日常查文档

用户问题:

> watch 和 watchEffect 有什么区别?

期望结果:

- 给出清晰对比
- 说明使用场景
- 引用 Vue 官方文档中对应章节
- 不输出无法验证的个人经验式结论

### 场景 2: 版本差异查询

用户问题:

> defineModel 怎么用?

期望结果:

- 说明 `defineModel` 的基本用法
- 标注该 API 的适用版本
- 如果用户当前选择 Vue 3.3，提示该版本不支持
- 引用 Vue 3.4+ 官方文档

### 场景 3: 升级迁移

用户问题:

> Vue2 的 filters 升到 Vue3 怎么改?

期望结果:

- 说明 Vue 3 移除了 filters
- 给出推荐替代方案，例如 methods 或 computed
- 引用 migration guide
- 标明这是迁移类问题，不是普通 API 用法问题

### 场景 4: 知识边界拒答

用户问题:

> React 的 useMemo 怎么用?

期望结果:

- 明确说明当前知识库暂不覆盖 React
- 不编造答案
- 引导用户切换到已支持的 Vue 生态问题

## 7. 功能范围

### 7.1 MVP 功能

| 功能 | 说明 | 优先级 |
|---|---|---|
| 文档问答 | 用户输入问题，系统基于官方文档回答 | P0 |
| 来源引用 | 答案中的关键结论带 `[1]`、`[2]` 等引用 | P0 |
| 引用详情 | 点击引用后展示原文片段、标题面包屑和精确到锚点的官方链接 | P0 |
| 框架选择 | 支持选择 Vue / Vite / Pinia / Vue Router | P1 |
| 版本选择 | 支持选择文档版本，并参与检索过滤 | P1 |
| 流式输出 | 回答以 SSE 流式生成，降低等待感 | P1 |
| 拒答机制 | 检索不到相关文档时明确拒答 | P0 |
| Markdown 渲染 | 支持列表、表格、代码块和代码高亮 | P1 |

### 7.2 后续功能

| 功能 | 说明 | 阶段 |
|---|---|---|
| 混合检索优化 | 向量检索 + BM25 + RRF + rerank | V2 |
| 多版本文档 | 同一框架支持多个版本文档 | V3 |
| 迁移模式 | 针对 Vue 2 到 Vue 3 的迁移问题优化 Prompt 和数据源 | V3.5 |
| 评测面板 | 展示检索命中率、引用准确率、延迟和成本 | V4 |
| 用户反馈 | 收集赞/踩和 bad case | V5 |
| 多轮追问 | 支持结合上下文继续追问 | V5 |

### 7.3 非功能需求

| 需求 | 要求 |
|---|---|
| 可用性 | 单点依赖(LLM / 精排服务)故障时服务可降级，不中断 |
| 防滥用 | IP 级限流(约 20 次/小时)+ 输入长度上限，保证 demo 长期可访问 |
| 成本 | 单次问答约 ¥0.005，月综合成本控制在 ¥30 以内(详见第 12 节) |
| 可分享性 | 框架/版本选择同步到 URL query，链接可直接分享复现 |

### 7.4 非目标

- 不做通用聊天机器人(off-topic 问题直接引导回文档问答)
- 不做代码生成 Agent
- 不在早期支持登录、付费、团队空间(V0–V4 无用户系统，靠限流防滥用)
- 不一次性覆盖所有前端框架(先做透 Vue 生态，架构预留扩展)
- 不在没有文档依据时生成看似合理的答案

## 8. 信息架构与交互设计

### 8.1 主界面

主界面由以下区域组成:

- 顶部栏:产品名、当前框架、当前版本
- 左侧或顶部控制区:框架选择器、版本选择器
- 中央问答区:用户问题和助手回答
- 底部输入区:问题输入框、发送按钮
- 右侧 citation 面板:展示引用片段、标题面包屑(heading path)、官方文档链接

### 8.2 回答结构

一次标准回答建议包含:

1. 直接结论
2. 关键解释
3. 示例或对比
4. 版本适用说明
5. 来源引用

示例结构:

```text
watch 适合监听明确的数据源，watchEffect 会自动追踪回调中访问到的响应式依赖。[1]

如果你需要精确控制监听对象，优先用 watch；如果依赖比较多，且希望自动收集依赖，可以用 watchEffect。[1]

适用版本: Vue 3.x。
```

### 8.3 关键交互细节

| 交互 | 体验要求 |
|---|---|
| 流式输出 | 回答逐 token 流出，输出中自动滚动到底部；用户上滑时立即停止自动跟随，不打断阅读 |
| 流式渲染容错 | 流到一半的代码块、表格等不完整 Markdown 也能正常渲染，不出现闪烁或破版 |
| citation 交互 | `[n]` 渲染为可点击徽标 → 侧栏展示片段原文并高亮 → 可跳转官方文档对应锚点；解析失败时降级为纯文本，不影响阅读 |
| 版本切换 | 切换框架/版本后状态同步到 URL；再次提问自动携带新的过滤条件 |
| 版本标注 | 回答末尾始终展示「适用版本」；当 API 与当前选择版本冲突时给出显著提示 |

## 9. 核心用户流程

### 9.1 普通问答流程

```text
打开页面
→ 选择框架和版本
→ 输入问题
→ 系统检索官方文档
→ 流式生成答案
→ 用户点击 citation 查看来源
→ 用户跳转官方文档验证
```

### 9.2 版本冲突流程

```text
用户选择 Vue 3.3
→ 提问 defineModel 用法
→ 系统检索当前版本
→ 当前版本无支持证据
→ 系统提示该 API 需要 Vue 3.4+
→ 建议用户切换版本或查看对应官方文档
```

### 9.3 拒答流程

```text
用户提出知识库外问题
→ 检索结果为空或置信度过低
→ 系统拒答
→ 告知当前覆盖范围
→ 建议换成 Vue 生态相关问题
```

### 9.4 异常与降级体验

服务故障时用户看到的不是报错，而是有明确说明的降级结果:

| 故障 | 用户侧体验 |
|---|---|
| 生成失败(LLM 超时/出错) | 自动重试并切换备用模型；仍失败则展示检索到的相关文档片段列表:「生成失败，这些是相关文档」，用户仍可自行查阅 |
| 精排服务异常 | 自动跳过精排，直接使用融合排序结果；回答质量轻微下降但服务不中断，用户无感知 |
| 检索零命中 | 明确拒答，并建议换个问法或切换版本，绝不让模型脱离文档作答 |
| 请求超限 | 提示已达限流阈值和恢复时间 |

## 11. 成功指标与评测

### 11.1 产品指标

- 用户能在 30 秒内获得可验证答案
- 主要答案都能提供官方来源引用
- 知识库外问题能够稳定拒答
- 版本相关问题能够明确提示适用版本

### 11.2 技术指标

- 检索命中率:golden chunk 进入 top-5 的比例(纯代码可算，最客观)
- 引用准确率:citation 是否真正支持对应结论
- 答案忠实度:回答是否完全基于检索片段(LLM-as-judge 按 rubric 打分)
- 延迟:P50 / P95 响应时间
- 成本:单次问答平均 LLM 调用成本

### 11.3 评测集构成

评测集为 30–50 题 JSONL，每题标注 golden chunk 与要点答案，覆盖四类问题:

| 题型 | 对应场景 | 说明 |
|---|---|---|
| 事实题 | 场景 1 | 文档中有明确答案的常规问题 |
| 代码用法题 | 场景 1 | 需要给出代码示例的 API 用法问题 |
| 版本差异题 | 场景 2/3 | **主打特色题型**，检验版本标注与版本冲突提示 |
| 应拒答题 | 场景 4 | 知识库外问题，检验拒答边界 |

**评测流程铁律:先跑 baseline，每完成一项优化(分块调参 / 混合检索 / rerank)重跑一次，记录前后对比表进 README。**

## 12. 成本与可持续性

产品以「长期可访问的免费 demo」为运营目标，成本压到奶茶钱:

| 项 | 用量估算 | 成本 |
|---|---|---|
| 文档建库 | Vue 生态全量文档约数万 chunk，一次性 | 本地 embedding ≈ ¥0；走 API 也 < ¥5 |
| 单次问答 | 检索免费 + rerank + LLM(约 4–6k token 上下文) | ≈ ¥0.005 / 次 |
| 月成本 | 开发调试 + 评测跑分 + 少量真实访问 | **< ¥30 / 月** |
| 托管 | 前端 Vercel 免费档 + 后端 Render/Railway 免费档 | ¥0 |

控制手段:IP 限流、上下文截断、回答长度上限、评测用缓存。

## 13. 风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| 文档版本难获取 | 多版本能力延期 | 先做单版本，后续按 git tag 或分支扩展 |
| citation 解析不稳定 | 用户难以验证答案 | 先采用简单编号引用，再逐步增强交互；解析失败降级为纯文本 |
| 检索质量不够 | 答案不准或拒答过多 | 引入 BM25、RRF、reranker 和评测集，量化驱动优化 |
| LLM 幻觉 | 产品可信度下降 | 强约束 Prompt，检索不到就拒答 |
| 部署成本超预期 | Demo 难长期运行 | 限流、缓存、免费托管、低价模型 |
