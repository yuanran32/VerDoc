# 暑假项目 · 技术方案(产品说明 + 技术栈 + 整体架构)

> 项目:「前端框架文档智能助手」(暂名,候选:DocLens / VerDoc / 文档罗盘)
> 本文档 = 产品说明书 + 技术选型 + 架构设计,与 `暑假项目-产品路线.md`(V0–V5 路线 + 8 周计划)配套使用。
> 最后更新:2026-07-05

## 第一部分 · 产品说明

### 1.1 一句话定位

面向前端开发者的**版本感知**文档问答助手:聚合前端框架官方文档,回答带**来源引用 + 版本标注**,不知道就明说。

### 1.2 目标用户与核心场景

**目标用户**:使用 Vue 生态(Vue3 / Vite / Pinia / Vue Router)的前端开发者,尤其是被版本差异坑过的人。

**四个典型用户故事**:

| # | 场景 | 用户问 | 产品答 | 对应版本 |
|---|---|---|---|---|
| 1 | 日常查文档 | "watch 和 watchEffect 有什么区别?" | 基于官方文档的准确回答 + 可点击的来源引用 | V0/V1 |
| 2 | 版本困惑 | "defineModel 怎么用?" | 回答 + 标注「Vue 3.4+ 才可用」,当前选的是 3.3 则明确提示 | V3 |
| 3 | 升级迁移 | "Vue2 的 filters 升到 Vue3 怎么改?" | 迁移建议 + breaking change 出处(migration guide) | V3.5/V5 |
| 4 | 知识边界 | "React 的 useMemo 怎么用?" | 诚实拒答:「当前知识库仅覆盖 Vue 生态」,不瞎编 | V1 |

### 1.3 三痛点 → 三解法

| 痛点 | 现状 | 本产品解法 |
|---|---|---|
| 文档版本混乱 | 搜到的答案不知道适用哪个版本,Vue2/Vue3 混着出 | **版本元数据过滤 + 回答带版本标注 + 版本切换 UI** |
| LLM 幻觉 | 通用 Chat 编造不存在的 API | **RAG 强约束:只基于检索片段回答,检索不到就拒答** |
| 答案无出处 | 无法验证,不敢直接用 | **Citation 溯源:每个结论可点击跳转官方文档原文** |

### 1.4 竞品与差异化(2026-07-05 调研)

| 竞品 | 性质 | 与本项目关系 |
|---|---|---|
| [kapa.ai](https://www.kapa.ai) | 商业,200+ 企业客户 | 版本感知是其招牌功能 → **证明该功能商业价值**;不开源 |
| [chat-langchain](https://github.com/langchain-ai/chat-langchain) | 开源,LangChain 官方 | 架构最佳参考(guardrails/citation/流式);无版本感知 |
| [kotaemon](https://github.com/Cinnamon/kotaemon) | 开源 RAG UI | 混合检索 + rerank 实现参考 |
| [OpenDocuments](https://github.com/joungminsung/OpenDocuments) | 开源自托管 | citation + 置信分参考 |

**差异化结论:版本感知(version-aware)是「商业已验证 + 开源无人做成一等公民」的空白点,本项目主打它。**

### 1.5 非目标(防止范围膨胀)

- ❌ 不做通用聊天机器人(off-topic 直接引导回文档问答)
- ❌ 不做代码生成 Agent(那是 pivot-workbench 和 V5 之后的事)
- ❌ 不追求一次覆盖多框架(先做透 Vue 生态,架构预留扩展)
- ❌ V0–V4 不做用户系统 / 登录 / 付费(demo 属性,限流防滥用即可)

## 第二部分 · 技术栈

### 2.1 总览

| 层 | 技术 | 选型理由 |
|---|---|---|
| 前端框架 | **Next.js 15(App Router)+ TypeScript** | 中大厂 React 主流栈,补齐规划中的 Next.js;RSC/流式友好 |
| 样式/组件 | **Tailwind CSS v4** + shadcn/ui(按需) | 与 ElowenBlog 技术延续;shadcn 是 React 生态事实标准 |
| Markdown 渲染 | react-markdown + remark-gfm + **shiki**(代码高亮) | 流式增量渲染;shiki 与 VS Code 同款高亮 |
| 流式通信 | **SSE**(fetch + ReadableStream 手动解析) | 单向流够用、比 WebSocket 简单;fetch 版支持 POST + 自定义头 |
| 后端框架 | **Python 3.12 + FastAPI + Pydantic v2** | 规划要补的 AI 服务层语言;异步 + 类型校验 + 自动文档 |
| Embedding | **bge-m3**(本地/硅基流动 API) | 中英双语强、免费或极便宜;备选 text-embedding-3-small |
| 向量库 | **Chroma**(开发)→ pgvector(上线可选) | Chroma 零配置快速迭代;pgvector 顺便练 Postgres |
| 关键词检索 | **rank_bm25**(轻量库) | 8 周项目不上 Elasticsearch,内存 BM25 足够 |
| Reranker | **bge-reranker-v2-m3**(本地/API) | 开源、效果好;备选 Cohere Rerank API |
| LLM | **DeepSeek / GLM / gpt-4o-mini**(OpenAI 兼容接口) | 便宜够用;代码统一走 OpenAI SDK,随时换模型 |
| 评测 | 自建评测脚本(RAGAS 思路)+ pytest | 命中率/忠实度/引用准确率,详见 3.7 |
| 部署 | 前端 **Vercel** / 后端 **Render 或 Railway**(免费档) | 已有 Vercel 经验;后端免费额度够 demo |
| 工程化 | pnpm / **uv**(Python 包管理)/ GitHub Actions CI | uv 是 2026 Python 事实标准,快 |

### 2.2 关键取舍(面试必讲)

1. **RAG 核心链路手写,不用 LlamaIndex 全家桶** —— 分块、检索、RRF、Prompt 组装自己写(每块都不难),面试能讲原理不露馅;仅在文档解析等边缘环节按需借库。
2. **混合检索是必需不是炫技** —— 前端文档满是 `defineModel`、`v-model` 这类符号化 API 名,纯向量检索对精确 token 匹配弱,BM25 补上;语义问题("怎么做双向绑定")靠向量。RRF 融合免调权。
3. **版本过滤用 pre-filter**(检索前按元数据过滤)而非检索后过滤 —— 保证 top-k 名额不被其他版本挤占,这是版本感知准确率的关键。
4. **SSE 而非 WebSocket** —— 问答是单向流,SSE 基于 HTTP,部署/重连/降级都简单。

### 2.3 成本预算(压到奶茶钱)

| 项 | 用量估算 | 成本 |
|---|---|---|
| Embedding 建库 | Vue 生态全量文档 ≈ 数万 chunk,一次性 | bge-m3 本地 ≈ ¥0;API 也 < ¥5 |
| 单次问答 | 检索免费 + rerank + LLM(约 4–6k token 上下文) | ≈ ¥0.005 / 次(DeepSeek) |
| 月成本(开发+demo) | 开发调试 + 评测跑分 + 少量真实访问 | **< ¥30 / 月** |
| 托管 | Vercel 免费档 + Render/Railway 免费档 | ¥0 |

控制手段:限流(IP 级)、上下文截断、回答长度上限、评测用缓存。

## 第三部分 · 整体架构

### 3.1 架构总览

```
═══════════════ 离线建库管线(文档更新时手动/CI 触发)═══════════════

 GitHub 文档仓库(vuejs/docs、vitejs/docs、pinia、router;不同 tag = 不同版本)
     │  git clone(直接拉 .md,不爬 HTML)
     ▼
 解析清洗 ──▶ 结构化分块 ──▶ 元数据标注 ──▶ 向量化(bge-m3)
 (剥 frontmatter    (标题感知,       {framework, version,      │
   /MDX 组件)        代码块不切断)     heading_path, url…}       ▼
                                              ┌─────────────┬──────────┐
                                              │ Chroma 向量库│ BM25 索引 │
                                              └─────────────┴──────────┘

═══════════════ 在线问答服务 ═══════════════

 用户 ──▶ Next.js 前端(流式 UI / citation 展示 / 版本切换)
             │ POST /api/chat(SSE)
             ▼
        FastAPI 服务层(参数校验 · 限流 · 日志 · token 成本统计)
             ▼
        ① 检索:版本 pre-filter ─▶ [向量召回 ∥ BM25 召回] ─▶ RRF 融合 ─▶ Rerank ─▶ Top-5
             ▼
        ② 生成:Prompt 组装(带编号片段)─▶ LLM 流式生成 ─▶ citation 解析([n] → chunk)
             ▼
 用户 ◀── SSE 事件流:token / citations / 版本标注 / done · error

═══════════════ 评测层(离线,每次改动后跑)═══════════════

 评测集(30–50 题 JSONL)──▶ run_eval.py ──▶ 检索命中率 · 忠实度 · 引用准确率 · 成本/延迟
```

### 3.2 七层模块职责

| 模块 | 职责 | 关键设计点 |
|---|---|---|
| 数据层 | 拉取/解析/分块文档 | 标题感知分块(按 `##` 层级),chunk 200–500 token 带 overlap,**代码块保持完整** |
| 索引层 | 向量库 + BM25 双索引 | 增量更新:文档变更只重建对应 chunk(参考 umbertogriffo/rag-chatbot) |
| 检索层 | 召回 + 融合 + 精排 | pre-filter → 双路各取 top-50 → RRF → rerank → top-5 |
| 生成层 | Prompt 组装 + LLM + citation | 强约束 system prompt:只基于片段回答、输出 [n] 引用标记、检索不到就拒答 |
| 服务层 | FastAPI:API / 限流 / 日志 | SSE 流式;错误分级降级(见 3.6) |
| 前端层 | 问答 UI | 流式增量 markdown 渲染、citation 点击定位、版本切换(URL 同步) |
| 评测/可观测层 | 评测集 + 指标 + 成本统计 | 每次优化前后跑分对比,数字进 README(简历核弹) |

### 3.3 数据模型:chunk 元数据 Schema(版本感知的核心)

```json
{
  "id": "vue@3.4:guide/essentials/watchers#watcheffect:0002",
  "text": "watchEffect() 允许我们自动跟踪回调的响应式依赖……",
  "framework": "vue",            // vue | vite | pinia | vue-router
  "version": "3.4",              // 来自 git tag / 分支
  "lang": "zh",                  // zh | en
  "source_path": "src/guide/essentials/watchers.md",
  "source_url": "https://cn.vuejs.org/guide/essentials/watchers#watcheffect",
  "heading_path": ["指南", "侦听器", "watchEffect()"],   // 面包屑,拼进 Prompt 增强上下文
  "chunk_type": "text"           // text | code | api-ref
}
```

设计要点:`source_url` 精确到 heading 锚点(citation 可点击直达);`heading_path` 随片段一起进 Prompt(LLM 更懂片段来源);`version` 支撑 pre-filter 与回答标注。

### 3.4 API 设计

| 接口 | 方法 | 说明 |
|---|---|---|
| `/api/chat` | POST(SSE) | 入参 `{query, framework?, version?, history?}`;事件流:`token` → `citations` → `done` / `error` |
| `/api/meta` | GET | 返回支持的框架列表 + 各框架可用版本(前端版本切换器数据源) |
| `/api/feedback` | POST | 👍/👎 + bad case 收集(V5,数据飞轮) |

### 3.5 前端关键实现点(前端岗面试弹药)

1. **流式渲染**:fetch ReadableStream 手动解析 SSE;token 缓冲合并(~60ms 批量 flush)防止高频 setState;消息列表 memo 分段,只重渲染最后一条。
2. **不完整 markdown 增量渲染**:流到一半的代码块/表格容错(未闭合 ``` 临时补全再渲染)。
3. **citation 交互**:`[n]` 渲染为可点击徽标 → 侧栏展示片段原文 + 高亮 → 跳官方文档锚点;解析失败降级为纯文本。
4. **版本切换**:框架/版本选择器,状态同步到 URL query(可分享);切换后重新提问自动带过滤条件。
5. **滚动跟随**:流式输出自动滚底,用户上滑立即停止跟随(交互细节分)。

### 3.6 错误处理与降级链

| 故障 | 降级策略 |
|---|---|
| LLM 超时/失败 | 重试 1 次 → 换备用模型 → 返回检索片段列表("生成失败,这些是相关文档") |
| Reranker 挂了 | 跳过精排,直接用 RRF 排序结果 |
| 检索 0 命中 | 明确拒答 + 建议换问法/换版本,**绝不让 LLM 裸答** |
| 滥用流量 | IP 限流(如 20 次/小时)+ 输入长度上限 |

### 3.7 评测体系(V4 核心,简历最值钱的部分)

- **评测集**:30–50 题 JSONL,四类题各占一定比例 —— 事实题 / 代码用法题 / **版本差异题**(主打特色)/ 应拒答题(边界)。每题标注 golden chunk 与要点答案。
- **指标**:
  - 检索命中率:golden chunk 是否进 top-5(纯代码可算,最客观)
  - 答案忠实度:LLM-as-judge 按 rubric 打分(答案是否被片段支持)
  - 引用准确率:citation 指向的片段是否真支持对应结论
  - 延迟 P50/P95 + 单题成本
- **流程铁律**:W6 先测 baseline → 每做一项优化(分块调参/混合检索/rerank)重跑 → 记录前后对比表进 README。
